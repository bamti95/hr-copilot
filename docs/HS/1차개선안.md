# 실험 개요

- **대상:** scorer(점수 평가) 노드
- **한 줄 목적:** (예: questioner 노드의 출력 토큰 최적화를 통한 전체 Latency 단축)
- **실험 유형**: 성능 / 비용 / 안정성 / 품질 / 모델 라우팅 / State 구조 개선

### 문제 현상

- scorer 노드가 약 **360초 실행 후 timeout으로 실패**
- questioner, driller, reviewer는 정상 수행되었으나 **최종 scorer 단계에서 전체 워크플로우 실패**
- reviewer에서 이미 수행된 평가를 scorer에서 다시 LLM으로 평가하는 **중복 구조 존재**

### 관측 데이터

| 항목 | 측정 데이터 | 비고 |
| --- | --- | --- |
| 전체 지연 시간 | ~360s (실패) | scorer에서 종료 |
| 문제 노드 지연 시간 | ~360s | 전체 중 100% |
| 평균 노드 지연 시간 | 30~50s | questioner / reviewer 기준 |
| 입력 토큰 | 매우 높음 | questions + answers + followups + reviews 전체 |
| 출력 토큰 | 발생 | LLM 재평가 |
| 호출 비용 | 발생 | scorer에서도 LLM 호출
아래 데이터를 모두 다시 LLM에 넣고 점수를 매겼음
questions + answers + followups + reviews |
| 성공률 | 낮음 | timeout 발생 |
| 에러 유형 | timeout | scorer 단계 |

# 실험 설계

> 이 섹션에서는 무엇을 바꾸고, 무엇은 유지했는지 분리해서 기록한다.
> 

### 변경 내용

| 구분 | AS-IS | TO-BE |
| --- | --- | --- |
| 점수화 방식 | LLM 기반 전체 재평가 | 규칙 기반 점수 계산 - 파이썬 로직으로 LLM 호출 없음 → 비용 X |
| 모델 | gpt-5-mini | 사용 안함 |
| 출력 형식 | LLM 응답 파싱 | Python dict 직접 생성 |
| State 전달 데이터 | questions / answers / followups / reviews 전체 전달 | 동일 State 사용하지만 LLM 전달 없음 |
| 파라미터 | temperature / max_tokens 존재 | 제거 |
| timeout / retry | timeout 발생 가능 | Python 로직 → timeout 없음 |

### 변인 통제

| 항목 | 유지 / 변경 | 내용 |
| --- | --- | --- |
| 모델 | 🟢 유지 | LLM 제거 |
| 프롬프트 | 🔴 제거 | rule-based 전환 |
| 테스트 입력 | 🟢 유지 | 동일 데이터 |
| temperature | 🔴 제거 | 사용 안함 |
| max tokens | 🔴 제거 | 사용 안함 |
| 실행 횟수 | 🟢 유지 | 동일 기준 |

### 설계 근거

- reviewer가 이미 품질 평가를 수행하므로 scorer는 **재평가가 아닌 집계 역할**이 적절
- LLM 호출 제거 시 **토큰/비용/timeout을 동시에 제거 가능**
- 점수는 정밀 평가가 아니라 **라우팅용 품질 신호**로 충분

# 회고

### 알게 된 점

- 병목은 모델 성능이 아니라 **불필요한 LLM 호출 구조**에서 발생했다
- scorer는 평가 노드가 아니라 **집계 노드로 역할을 재정의하는 것이 적절**
- LLM을 줄이는 것이 가장 강력한 성능 개선 방법

### 예상과 달랐던 점

- scorer가 핵심 병목일 것이라 예상하지 못했음
- reviewer 이후 단계에서 실패가 발생할 것이라고 생각하지 않았음
- 모델 변경 없이도 **구조 개선만으로 문제 해결 가능**

### 추가로 확인해야 할 점

- 규칙 기반 점수와 LLM 평가 간 품질 차이 비교
- 특정 케이스에서 점수 왜곡 여부 확인
- 향후 hybrid scoring (rule + LLM) 필요성 검토

### 한 줄 정리

> 
> 
> 
> scorer 노드의 360초 timeout 문제를 관측하고, reviewer와의 중복 LLM 평가 및 과도한 State 전달을 원인으로 가설 설정했다.
> 
> LLM 호출을 제거하고 규칙 기반 점수 집계로 전환한 결과, 비용과 latency를 제거하고 안정성을 확보했으며,
> 
> 다만 정성 평가 nuance 감소를 감안해 점수는 라우팅 신호로 활용하기로 결정했다.
>
