# interview_graph LangGraph 분석 — AS-IS 및 partial_completed 원인 분석

---

## 1. 문서 목적

본 문서는 `backend/ai/interview_graph` 기준으로 현재 LangGraph 실행 구조, 각 노드의 상세 데이터 흐름, `PARTIAL_COMPLETED` 상태가 반복 발생하는 원인과 구조적 메커니즘을 정리한다.

---

## 2. 전체 실행 그래프 구조

```text
CandidateInterviewPrepInput
 └─▶ build_state
      └─▶ analyzer
           └─▶ questioner  ◀──────────────────────────┐
                └─▶ selector_lite                      │ (↻ 재시도)
                     └─▶ predictor                     │
                          └─▶ driller  ◀──────────────┤
                               └─▶ reviewer            │
                                    └─▶ scorer         │
                                         └─▶ route_after_scoring
                                              ├── retry_questioner ──▶ questioner (루프)
                                              ├── retry_driller ────▶ driller (루프)
                                              └── selector
                                                   └─▶ final_formatter
                                                        └─▶ QuestionGenerationResponse
```

---

## 3. 노드별 상세 데이터 흐름

### 3.1 build_state

| 구분 | 필드 |
|------|------|
| **IN** | `documents`, `session_id`, `target_job`, `difficulty_level`, `candidate_id`, `candidate_name` |
| **OUT** | `candidate_context` (병합 텍스트), `retry_count=0`, `max_retry_count=3` |
| **LLM** | 없음 |

`_merge_document_text()`가 세션 정보와 모든 문서의 `extracted_text`를 최대 18,000자 제한으로 병합하여 `candidate_context`를 구성한다. 문서에 추출 텍스트가 없어도 노드 자체는 실패하지 않으나, 이후 품질에 직접 영향을 준다.

---

### 3.2 analyzer

| 구분 | 필드 |
|------|------|
| **IN** | `candidate_context`, `prompt_profile` |
| **OUT** | `document_analysis` (DocumentAnalysisOutput), `llm_usages` |
| **LLM** | gpt-5-mini |

`DocumentAnalysisOutput` 구조:
- 강점 / 약점 / 리스크 포인트
- 문서 근거 (evidence)
- 직무 적합성 (job_fit)
- 면접 검증 포인트

---

### 3.3 questioner

| 구분 | 필드 |
|------|------|
| **IN** | `candidate_context`, `document_analysis`, `target_question_ids`, `retry_feedback`, `human_action`, `additional_instruction`, `prompt_profile` |
| **OUT** | `questions` (8개 또는 부분 재생성 N개), `target_question_ids=[]`, `human_action=None`, `llm_usages` |
| **LLM** | gpt-5-mini |

**부분 재생성 로직:**

```python
if human_action == "regenerate_question" and target_question_ids:
    # 대상 ID 수만큼 새 질문 생성
    question_count_instruction = f"질문은 정확히 {len(target_question_ids)}개 생성하세요."
    # 생성 후 기존 questions 목록에서 동일 ID 위치에 교체
    questions = _replace_questions_by_id(existing_questions, new_questions, target_question_ids)
```

`_replace_questions_by_id()`는 기존 질문 목록에서 `target_question_ids`에 해당하는 위치만 새 질문으로 교체하고, 나머지는 유지한다.

---

### 3.4 selector_lite

| 구분 | 필드 |
|------|------|
| **IN** | `questions` (8개) |
| **OUT** | `questions` (5개) |
| **LLM** | 없음 (규칙 기반) |

`_select_question_candidates()` 정렬 기준 (우선순위 순):
1. 리뷰 승인 여부 (현 시점은 리뷰 없으므로 False)
2. 점수 (현 시점은 0)
3. `document_evidence` 존재 여부
4. 리스크 질문 여부 (`category == RISK` 또는 `risk_tags` 존재)

카테고리 다양성 보장: 리스크 질문을 먼저 1개 확보하고, 이후 중복 카테고리를 피하면서 선별한다.

**토큰 절감 목적:** 이 노드가 없으면 8개 전부에 대해 `predictor`, `driller`, `reviewer`가 실행되어 비용이 1.6배 증가한다.

---

### 3.5 predictor

| 구분 | 필드 |
|------|------|
| **IN** | `candidate_context` (최대 6,000자), `document_analysis`, `questions` (5개) |
| **OUT** | `answers` (PredictedAnswer 5개), `llm_usages` |
| **LLM** | gpt-5-mini |

`PredictedAnswer` 구조:
- `question_id`
- `predicted_answer` (최대 300자 클리핑)
- `predicted_answer_basis` (최대 160자 클리핑)
- `answer_confidence`: high / medium / low
- `answer_risk_points`

실패 시 `_fallback_answer()`를 사용하고 `node_warnings`에 기록 → `partial_completed` 조건 발동.

---

### 3.6 driller

| 구분 | 필드 |
|------|------|
| **IN** | `questions`, `answers`, `document_analysis`, `retry_feedback` |
| **OUT** | `follow_ups` (FollowUpQuestion 5개), `llm_usages` |
| **LLM** | gpt-5-mini |

`FollowUpQuestion` 구조:
- `question_id`
- `follow_up_question`
- `follow_up_basis`
- `drill_type`: ROLE_VERIFICATION / METRIC_VERIFICATION / DECISION_REASONING / FAILURE_RECOVERY / COLLABORATION / RISK_RESPONSE / OTHER (+ 한국어 병용 허용)

실패 시 fallback (`"방금 답변하신 내용 중 직접 수행한 범위를..."`)으로 대체.

---

### 3.7 reviewer

| 구분 | 필드 |
|------|------|
| **IN** | `target_job`, `prompt_profile`, `questions`, `answers`, `follow_ups` |
| **OUT** | `reviews` (ReviewResult 5개), `llm_usages` |
| **LLM** | gpt-5-mini |

`ReviewResult` 구조:
- `question_id`
- `status`: `approved` / `needs_revision` / `rejected`
- `reason`: 판정 사유
- `reject_reason`: 반려 코드
- `recommended_revision`: 수정 권고

`approved_count < 5` 조건의 핵심 원인 노드. reviewer가 `needs_revision` 또는 `rejected`를 반환하면 이후 `scorer → router → partial_completed` 경로로 이어진다.

---

### 3.8 scorer (규칙 기반)

| 구분 | 필드 |
|------|------|
| **IN** | `questions`, `reviews`, `answers`, `follow_ups` |
| **OUT** | `scores` (ScoreResult 5개), `review_summary`, `llm_usages=[]` |
| **LLM** | 없음 (개선 완료) |

**점수 계산 공식:**

```python
REVIEW_SCORE_BASE = {"approved": 35, "needs_revision": 20, "rejected": 0}

score = REVIEW_SCORE_BASE[review_status]  # 기본 35 / 20 / 0
score += 20  # document_evidence 존재 시
score += 10  # evaluation_guide 존재 시
score += 10  # 리스크 질문 여부
score += 10  # competency_tags 존재 시
score += 10  # follow_up_question 존재 시 (없으면 -10)
score += 5   # answer_confidence == "high"
score -= 15  # 중복 질문 (duplicate_question_ids)
score -= 15  # reject_reason 존재 시
score = clamp(0, 100)
```

`LOW_SCORE_THRESHOLD = 80` 미만인 질문 ID는 `low_score_question_ids`로 수집.

`review_summary` 주요 출력:
- `approved_count`, `rejected_count`
- `avg_score`, `low_score_question_ids`, `low_score_count`
- `follow_up_issue_question_ids`, `question_rewrite_question_ids`
- `quality_issues` (플래그 목록)

---

### 3.9 route_after_scoring

| 구분 | 내용 |
|------|------|
| **IN** | `review_summary`, `retry_count`, `max_retry_count`, `questioner_retry_count`, `driller_retry_count` |
| **분기 결과** | `retry_questioner` / `retry_driller` / `selector` |
| **LLM** | 없음 |

**분기 우선순위 (router.py 기준):**

```text
1. retry_count >= max_retry_count  →  selector (강제 종료)
2. follow_up_issue 존재 AND driller_retry_count < max  →  retry_driller
3. low_score_count == 1~2 AND questioner_retry_count < max  →  retry_questioner (부분)
4. approved_count == 0 AND questioner_retry_count < max  →  retry_questioner (전체)
5. rejected_count > 0 AND questioner_retry_count < max  →  retry_questioner
6. needs_revision 존재 AND questioner_retry_count < max  →  retry_questioner
7. 그 외  →  selector
```

---

### 3.10 retry_questioner / retry_driller

**retry_questioner:**

```python
# PARTIAL_RETRY_MAX_LOW_SCORE_COUNT = 2
if 0 < len(low_score_question_ids) <= 2:
    # 해당 ID만 재생성 (부분 재생성)
    target_question_ids = low_score_question_ids
    human_action = "regenerate_question"
else:
    # 전체 재생성
    target_question_ids = []

retry_count += 1
questioner_retry_count += 1
retry_feedback = _build_retry_feedback(state)  # 품질 이슈 JSON
```

**retry_driller:**
```python
retry_count += 1
driller_retry_count += 1
retry_feedback = _build_retry_feedback(state)
```

---

### 3.11 selector

| 구분 | 필드 |
|------|------|
| **IN** | `questions`, `reviews`, `scores` |
| **OUT** | `questions` (최종 5개) |
| **LLM** | 없음 (규칙 기반) |

`selector_lite`와 동일한 `_select_question_candidates()` 사용하되, 이번에는 `reviews_by_id`와 `scores_by_id`를 함께 전달하여 리뷰 승인 여부와 점수를 정렬에 반영한다.

---

### 3.12 final_formatter

| 구분 | 필드 |
|------|------|
| **IN** | `questions` (5개), `answers`, `follow_ups`, `reviews`, `scores`, `document_analysis`, `retry_count`, `node_warnings` |
| **OUT** | `final_response` (QuestionGenerationResponse) |
| **LLM** | 없음 |

**`is_partial` 판단 조건:**

```python
is_partial = (
    not has_extracted_text        # ① 문서 추출 텍스트 없음
    or len(items) < 5             # ② 최종 질문 수 부족
    or approved_count < 5         # ③ 승인 질문 수 부족  ★ 주요 원인
    or retry_count >= max_retry_count  # ④ 재시도 한계 도달
    or bool(node_warnings)        # ⑤ 노드 경고 존재 (fallback 사용 시)
)

status = "partial_completed" if is_partial else "completed"
```

## 4. PARTIAL_COMPLETED 반복 발생 — 원인 분석

### 4.1 문제 현상

| 항목 | 내용 |
|------|------|
| 실행 결과 | `partial_completed` |
| 저장 상태 | `PARTIAL_COMPLETED` (`interview_sessions.question_generation_status`) |
| 질문 생성 여부 | 질문은 저장됨 (완전 실패 아님) |
| 주요 의심 조건 | `approved_count < 5` |

---

### 4.2 원인 후보별 구조 분석

| ID | 원인 후보 | 메커니즘 | 발생 경로 |
|----|-----------|----------|-----------|
| H1 | 승인 질문 수 부족 | reviewer가 `needs_revision` 또는 `rejected` 반환 | `reviewer → scorer → router → selector → final_formatter (approved_count < 5)` |
| H2 | 재시도 횟수 초과 | `max_questioner_retry_count` 기본 1회, 재시도 후에도 미달 | `retry_questioner → questioner → ... → scorer → router → selector → partial` |
| H3 | fallback 발생 | `predictor`, `driller`, `reviewer` LLM 호출 실패 | `node_warnings` 발생 → `is_partial=True` |
| H4 | 질문 수 부족 | `selector` 이후 5개 미만 선별 | `final_formatter (len(items) < 5)` |
| H5 | 문서 텍스트 없음 | `extracted_text` 전부 비어 있음 | `build_state → has_extracted_text=False → final_formatter` |

---

### 4.3 가장 높은 빈도 원인: H1 (approved_count < 5)

현재 `reviewer`는 5개 질문 각각에 대해 `approved` / `needs_revision` / `rejected`를 반환한다. 5개 중 단 1개라도 `approved`가 아니면 `approved_count < 5`가 되고, 이것만으로도 `partial_completed`가 된다.

```text
질문 5개 중:
  - approved: 4개
  - needs_revision: 1개
  → approved_count = 4 < 5
  → is_partial = True
  → status = "partial_completed"
```

router가 `retry_questioner`로 보내 재생성을 시도하지만, `max_questioner_retry_count=1`이므로 1회 재시도 후 개선이 없으면 `selector`로 넘어가고 동일하게 `partial_completed`가 된다.

---

### 4.4 구조적 흐름 요약

```text
questioner (8개 생성)
  ↓
selector_lite (5개 선별)
  ↓
predictor → driller → reviewer
  ↓
  [reviewer: approved < 5]
  ↓
scorer → review_summary 생성
  ↓
route_after_scoring
  ├── [retry_count < max] → retry_questioner → questioner (재생성)
  │       ↓ (재시도 1회 후도 approved < 5)
  │   selector → final_formatter → partial_completed  ★
  └── [retry_count >= max] → selector → final_formatter → partial_completed  ★
```

---

### 4.5 scorer 개선과 현재 문제의 차이

| 구분 | scorer timeout 개선 (완료) | partial_completed 문제 (현재) |
|------|---------------------------|-------------------------------|
| 문제 성격 | LLM 호출 timeout / 비용 | 품질 승인 조건 미달 |
| 직접 원인 | scorer LLM 중복 평가 | `final_formatter`의 완료 조건 |
| 개선 방향 | LLM scorer 제거 → 규칙 기반 집계 | 완료 조건 완화 또는 품질 보정 전략 개선 |
| 현재 상태 | **개선 완료** | 원인 분석 완료, 후속 실험 필요 |

## 5. 개선 방향 후보

### 5.1 완료 조건 완화 (단기)

현재 `approved_count < 5`를 절대 조건으로 사용하는 대신, 평균 점수 임계값이나 `approved + needs_revision` 합산 기준으로 완화하는 방안.

```python
# 예시: 승인 + 수정 필요 합산 >= 5이면 completed로 판단
effective_count = approved_count + needs_revision_count
is_partial = effective_count < 5 or avg_score < 60 or ...
```

### 5.2 재시도 횟수 확대 (단기)

`max_questioner_retry_count`를 1 → 2 또는 3으로 늘려 품질 보정 기회를 늘리는 방안. 다만 비용과 지연시간 증가 트레이드오프가 있다.

### 5.3 후보 질문 수 확대 (중기)

현재 8개 → 10~12개로 늘려 `selector_lite`의 선별 풀을 넓히면, 품질이 높은 질문이 최종 5개에 포함될 확률이 높아진다.

### 5.4 외부 평가자 노드 추가 (중기)

현재 생성자(questioner)와 평가자(reviewer)가 동일 파이프라인 내에서 순차 실행된다. 독립적인 시스템 프롬프트를 사용하는 별도 `external_reviewer` 노드 추가 시 제3자 검수 관점을 도입할 수 있다.

### 5.5 비용 개선 전후 비교 지표 (중기)

현재 노드별 사용량은 기록되나, `partial_completed` 비율, 재시도 횟수별 평균 비용, 후보자 1인당 비용 등 개선 효과를 직접 측정하는 지표가 부족하다.

```text
       approved_count < 5 가 가장 빈번하게 트리거됨

```
