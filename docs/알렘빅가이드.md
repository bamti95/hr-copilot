# Alembic 운영 가이드

## 1. 목적

이 프로젝트의 Alembic 설정 목적은 다음과 같습니다.

- SQLAlchemy `models` 변경 사항을 기준으로 DB 스키마 변경 이력을 관리
- 컬럼 추가/수정/삭제를 직접 SQL로 관리하지 않고 migration 파일로 관리
- 앱 실행 시 자동으로 테이블을 생성/수정하지 않고, 명시적인 migration 명령으로 반영

즉, 목표는 아래 흐름입니다.

```text
Model 변경
-> Alembic migration 생성
-> migration 검토
-> DB 반영
```

## 3. 현재 동작 방식

현재는 아래 방식입니다.

### 되는 것

- 모델 컬럼 추가
- Alembic이 변경점 감지
- migration 파일 생성
- 명령으로 DB 반영

### 안 되는 것

- 서버 실행만으로 컬럼 자동 반영
- 모델 수정 즉시 DB 자동 변경

즉, 현재 구조는 다음과 같습니다.

```text
자동 반영 X
반자동 관리 O
```

정확히는:

```text
Model 수정
-> alembic revision --autogenerate
-> alembic upgrade head
```

## 4. 핵심 설정 설명

### 4.1 `alembic.ini`

역할:

- Alembic 실행 기본 설정
- migration 스크립트 위치 지정

주요 설정:

```ini
[alembic]
script_location = alembic
prepend_sys_path = .
```

의미:

- `backend/alembic` 폴더를 migration 루트로 사용
- 프로젝트 import 경로를 현재 디렉터리 기준으로 맞춤

---

### 4.2 `env.py`

역할:

- Alembic이 어떤 DB에 연결할지 결정
- 어떤 metadata를 기준으로 비교할지 결정

현재 핵심 포인트:

- `DATABASE_SYNC_URL` 사용
- `target_metadata = Base.metadata`
- `compare_type=True`
- `compare_server_default=True`

의미:

- 모델과 실제 DB를 비교 가능
- 타입 변경도 비교 가능
- 기본값 변경도 비교 가능

---

### 4.3 `config.py`

역할:

- 앱/마이그레이션 공통 DB 설정 관리

현재 추가된 속성:

- `DATABASE_URL`
  - 앱 비동기용
- `DATABASE_SYNC_URL`
  - Alembic 동기용

예시:

```python
@property
def DATABASE_SYNC_URL(self) -> str:
    return (
        f"postgresql://{self.DB_USER}:{self.DB_PASSWORD}"
        f"@{self.DB_HOST}:{self.DB_PORT}/{self.DB_NAME}"
    )
```

---

### 4.4 `database.py`

역할:

- 앱 런타임 DB 연결 관리

현재 상태:

- `create_all()` 제거
- 앱 시작 시 스키마 생성/수정하지 않음
- 연결 확인만 수행

의미:

- DB 구조 변경은 Alembic만 담당
- 앱 실행이 스키마를 몰래 바꾸지 않음

## 5. 기존 DB와 새 DB 차이

### 5.1 새 DB

새 DB는 테이블이 없으므로 아래 명령으로 전체 생성 가능합니다.

```powershell
cd backend
uv run python -m alembic -c alembic.ini upgrade head
```

---

### 5.2 기존 DB

기존 DB는 이미 테이블이 있는데 Alembic 이력이 없을 수 있습니다.

이 경우 바로 `upgrade head`를 하면 baseline migration이 테이블 생성 시도를 할 수 있습니다.

그래서 기존 DB는 먼저 `stamp`가 필요합니다.

```powershell
cd backend
uv run python -m alembic -c alembic.ini stamp 20260417_000001
uv run python -m alembic -c alembic.ini upgrade head
```

의미:

- `stamp`: 실제 SQL 실행 없이 "이 리비전까지 반영된 것으로 기록"
- `upgrade`: 이후 migration만 실행

## 6. 모델 변경 시 처리 절차

예를 들어 `Document` 모델이 아래처럼 변경되었다고 가정합니다.

### 변경 전

```python
class Document(Base, AuditBase):
    __tablename__ = "document"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    document_type: Mapped[str] = mapped_column(String(30), nullable=False)
    title: Mapped[str] = mapped_column(String(255), nullable=False)
    file_path: Mapped[str] = mapped_column(String(500), nullable=False)
    candidate_id: Mapped[int] = mapped_column(ForeignKey("candidate.id"), nullable=False)
    extracted_text: Mapped[str | None] = mapped_column(Text, nullable=True)
    extract_status: Mapped[str] = mapped_column(String(20), nullable=False, default="PENDING")
```

### 변경 후

```python
class Document(Base, AuditBase):
    __tablename__ = "document"

    id: Mapped[int] = mapped_column(Integer, primary_key=True, autoincrement=True)
    document_type: Mapped[str] = mapped_column(String(30), nullable=False)
    title: Mapped[str] = mapped_column(String(255), nullable=False)

    original_file_name: Mapped[str] = mapped_column(String(255), nullable=False)
    stored_file_name: Mapped[str] = mapped_column(String(255), nullable=False)
    file_path: Mapped[str] = mapped_column(String(500), nullable=False)
    file_ext: Mapped[str | None] = mapped_column(String(20), nullable=True)
    mime_type: Mapped[str | None] = mapped_column(String(100), nullable=True)
    file_size: Mapped[int | None] = mapped_column(BigInteger, nullable=True)

    candidate_id: Mapped[int] = mapped_column(ForeignKey("candidate.id"), nullable=False)
    extracted_text: Mapped[str | None] = mapped_column(Text, nullable=True)
    extract_status: Mapped[str] = mapped_column(String(20), nullable=False, default="PENDING")
```

이 경우 목표는 테이블 재생성이 아니라 아래입니다.

- `document` 기존 테이블 유지
- 새 컬럼만 추가
- 필요한 경우 nullable/default 조정

현재 Alembic 구조는 이 방향에 맞습니다.

## 7. 실행 명령어

### 7.1 현재 상태 확인

현재 적용 revision 확인:

```powershell
cd backend
uv run python -m alembic -c alembic.ini current
```

최신 head 확인:

```powershell
cd backend
uv run python -m alembic -c alembic.ini heads
```

이력 확인:

```powershell
cd backend
uv run python -m alembic -c alembic.ini history
```

---

### 7.2 migration 생성

모델 변경 후:

```powershell
cd backend
uv run python -m alembic -c alembic.ini revision --autogenerate -m "update document table"
```

---

### 7.3 migration 반영

```powershell
cd backend
uv run python -m alembic -c alembic.ini upgrade head
```

---

### 7.4 한 단계 롤백

```powershell
cd backend
uv run python -m alembic -c alembic.ini downgrade -1
```

---

### 7.5 특정 revision 내용 확인

```powershell
cd backend
uv run python -m alembic -c alembic.ini show 20260417_000002
```

## 9. 주의사항

- `autogenerate`는 완전 자동이 아닙니다.
- 컬럼 추가는 잘 잡히는 편입니다.
- 아래는 반드시 사람이 검토해야 합니다.

꼭 검토할 항목:

- 컬럼 rename
- 컬럼 삭제
- enum 변경
- 데이터 이관
- 복합 제약조건 변경
- 기존 데이터에 대한 `NOT NULL` 변경

즉, migration 생성 후에는 `versions` 폴더의 파일을 꼭 확인해야 합니다.

## 10. 결론

현재 Alembic 구성은 아래 목표에 적합합니다.

- 모델 변경 기준으로 DB 스키마 관리
- 컬럼 추가/수정 migration 생성
- 명시적인 명령으로 DB 반영

다만 아래처럼 이해하셔야 정확합니다.

```text
모델 변경 시 자동 감지 O
앱 실행만으로 자동 반영 X
명령 실행으로 반영 O
```

실무 기준 최종 흐름은 아래로 정리됩니다.

```text
1. 모델 수정
2. uv run python -m alembic -c alembic.ini revision --autogenerate -m "..."
3. 생성된 migration 검토
4. uv run python -m alembic -c alembic.ini upgrade head
```
