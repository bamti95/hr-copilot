# 일괄등록 사전 면접자 선별 설계안

## 1. 컨펌

현재 코드베이스 기준으로 “문서 일괄등록 미리보기 단계에서 지원자 기본정보와 문서 데이터를 기반으로 1차 면접자 선별”을 붙이는 방향은 가능하고, 구조적으로도 적합하다.

이유:

- `DocumentBulkImportService`가 이미 preview job에서 파일 staging, 텍스트 추출, 지원자 프로필 추론, 중복 검사, row 상태 산정을 수행한다.
- preview 결과는 `AiJob.result_payload.rows`에 저장되고, `CandidateDetailPage.tsx`가 polling으로 조회해 `CandidateDetailModal.tsx`에 전달한다.
- 즉, 확정 저장 전에도 row 단위로 충분한 데이터가 존재하므로 `screening_preview` 결과를 row에 붙이면 일괄등록된 지원자 후보를 저장 전에 선별/검토할 수 있다.
- 확정 저장 시에는 HR이 선택한 row만 candidate/document로 생성하므로, “선별된 후보만 등록” 또는 “전체 등록하되 선별 결과 보존” 중 하나를 선택할 수 있다.

권장 UX는 **전체 preview row를 보여주되, AI 추천 후보를 기본 선택 상태로 제안**하는 방식이다. AI가 자동으로 탈락 처리하거나 저장 대상에서 강제 제외하면 위험하므로, 최종 선택은 HR 담당자가 한다.

## 2. 현재 흐름

현재 문서 기반 일괄등록 흐름:

```text
ZIP/다중 파일 업로드
→ AiJob(DOCUMENT_BULK_IMPORT) 생성
→ 파일 staging
→ 문서 텍스트 추출
→ 지원자 기본정보 추론
→ 중복/필수값 검증
→ preview row 생성
→ CandidateDetailModal에서 미리보기 표시
→ HR이 선택 row 확정
→ candidate/document 저장
```

추가할 흐름:

```text
문서 텍스트 추출
→ 지원자 기본정보 추론
→ 사전 면접자 선별 평가
→ screening_preview를 preview row에 저장
→ 추천/보류/비추천 상태를 미리보기에서 표시
→ 추천 후보를 기본 선택
→ HR이 최종 선택 row 확정
→ candidate/document 저장
→ 저장된 후보 상세/목록에서 선별 결과 조회
```

## 4. 선별 기준점

MVP에서는 LLM만 믿지 않고, deterministic score와 LLM 요약을 결합한다. 점수는 100점 만점이며, 자동 합격/불합격이 아니라 추천 우선순위다.

### 4.1 기본 산식

```text
screening_score =
profile_completeness_score       최대 20
+ document_readiness_score       최대 20
+ job_fit_signal_score           최대 30
+ evidence_quality_score         최대 20
+ risk_adjustment_score          -20 ~ +10
```

| 항목 | 배점 | 설명 |
|---|---:|---|
| 프로필 완성도 | 20 | 이름, 이메일/전화번호, 직무, 경력 수준 등 기본정보가 충분한지 |
| 문서 준비도 | 20 | 이력서/경력기술서/포트폴리오 등 핵심 문서가 있고 추출 품질이 좋은지 |
| 직무 적합 신호 | 30 | target_job과 문서 내 경험/기술/성과 키워드가 연결되는지 |
| 근거 품질 | 20 | 성과 수치, 프로젝트 역할, 기간, 도구, 결과가 구체적인지 |
| 리스크 조정 | -20~+10 | 중복 지원자, 추출 실패, 근거 부족, 민감정보 노출, 과장 가능성 등 |

### 4.2 추천 상태

| recommendation | 점수 기준 | 의미 | UI 표시 |
|---|---:|---|---|
| `RECOMMEND` | 75점 이상 | 면접 후보로 우선 검토 | 추천 |
| `HOLD` | 55~74점 | 추가 검토 필요 | 보류 |
| `NOT_RECOMMENDED` | 55점 미만 | 현재 근거로는 우선순위 낮음 | 비추천 |
| `NEEDS_REVIEW` | 점수와 무관 | 추출 실패/필수정보 부족/중복 등으로 사람 검토 필요 | 검토 필요 |

`READY` row라도 screening은 `HOLD`일 수 있고, `NEEDS_REVIEW` row라도 문서 근거가 좋아 `RECOMMEND`가 될 수 있다. 등록 가능 여부와 면접 추천 여부는 다른 축으로 봐야 한다.

## 6. 저장 모델 설계

preview 단계의 선별 결과는 `ai_job.result_payload`에 들어가므로 별도 테이블 없이 저장할 수 있다. 현재 `AiJob` 모델은 이미 다음 필드를 가지고 있어 사전 선별 결과 저장에도 재사용 가능하다.

```text
AiJob
- job_type
- status
- target_type
- candidate_id
- parent_job_id
- request_payload
- result_payload
- requested_by
- started_at
- completed_at
```

따라서 MVP에서는 **신규 테이블을 추가하지 않는 방식**을 권장한다.

### 6.1 Preview 단계

문서 일괄등록 preview job 하나가 여러 row의 선별 결과를 가진다.

```text
ai_job
- job_type = DOCUMENT_BULK_IMPORT
- target_type = BULK_IMPORT
- result_payload.rows[].screening_preview
```

이 방식은 현재 `CandidateDetailPage.tsx`의 polling 구조와 그대로 맞는다.

### 6.2 확정 저장 후

확정 저장으로 candidate가 생성되면, row별 `screening_preview`를 후보자 단위 `AiJob`으로 복사한다.

```text
ai_job
- job_type = CANDIDATE_RANKING
- status = SUCCESS
- target_type = CANDIDATE
- target_id = candidate.id
- candidate_id = candidate.id
- parent_job_id = document_bulk_preview_job.id
- request_payload = {
    "source": "DOCUMENT_BULK_IMPORT_PREVIEW",
    "preview_row_id": "..."
  }
- result_payload = {
    "screening_preview": {...},
    "decision_status": "PENDING"
  }
```

이렇게 하면 별도 `candidate_screening_result` 테이블 없이도 `CandidateDetailPage.tsx`에서 최신 선별 결과를 조회할 수 있다.

조회 조건 예:

```text
candidate_id = {candidate_id}
job_type = CANDIDATE_RANKING
target_type = CANDIDATE
status = SUCCESS
deleted_at IS NULL
ORDER BY created_at DESC
최신 job부터 result_payload.screening_preview가 있는 항목을 선택
```

### 6.3 별도 테이블이 필요한 시점

아래 요구가 생기면 그때 별도 `candidate_screening_result` 모델을 추가한다.

- 선별 결과를 후보자 목록에서 고성능 필터/정렬해야 함
- 추천/보류/제외 결정 이력을 여러 번 남겨야 함
- 선별 run 단위 통계가 필요함
- HR 담당자별 decision workflow가 필요함
- recommendation, score, decision_status를 DB 컬럼으로 직접 조회해야 함

### 보류 모델: candidate_screening_result

현재 MVP에서는 생성하지 않는다. 운영 고도화 시 아래 형태로 분리할 수 있다.

```text
candidate_screening_result
- id
- candidate_id
- source_type              # BULK_PREVIEW, MANUAL_SCREENING
- source_job_id            # ai_job.id
- preview_row_id
- recommendation           # RECOMMEND, HOLD, NOT_RECOMMENDED, NEEDS_REVIEW
- score
- confidence
- summary
- fit_reasons              # JSONB
- risk_factors             # JSONB
- missing_evidence         # JSONB
- interview_focus          # JSONB
- suggested_next_action
- score_breakdown          # JSONB
- evidence_refs            # JSONB
- raw_output               # JSONB
- decision_status          # PENDING, CONFIRMED, HELD, EXCLUDED
- decided_by
- decided_at
- created_at
- updated_at
```

`candidate_screening_run`은 배치 이력/요약이 필요해지는 시점에 추가한다.

## 7. 선별 방식 단계화

### 1단계: 휴리스틱 기반 빠른 선별

preview 처리 시간과 비용을 줄이기 위해 먼저 rule-based score를 계산한다.

입력:

- `candidate` 추론값
- `extracted_profile.summary`
- 문서 타입별 존재 여부
- `extract_quality_score`
- `extracted_text_preview`
- 중복 여부
- 필수 필드 누락 여부

장점:

- 빠름
- 비용 없음
- 일괄등록 preview UX를 막지 않음

### 2단계: LLM 기반 요약/근거 보강

OpenAI API 키가 있고 문서 텍스트가 충분한 row에 한해 LLM으로 다음을 보강한다.

- 직무 적합 근거
- 리스크 요인
- 면접 검증 포인트
- 추천 사유 문장

실패 시 휴리스틱 결과만 사용하고 `warnings`에 기록한다.

### 3단계: RAG 기반 공정성/직무 기준 주입

`RAG_채용절차법_공정성_설계안.md`의 `knowledge_chunk`가 준비되면 다음 기준을 함께 적용한다.

- 직무 무관 개인정보를 판단 근거에서 제외
- 출신지역/혼인/가족관계/신체조건 등 민감 신호 경고
- 공정 채용 기준에 위배될 수 있는 판단 표현 방지
- 채용공고/JD 분석 결과가 있으면 target_job 기준으로 함께 주입

## 8. UI 설계

### 8.1 일괄등록 미리보기 테이블

`CandidateDetailModal.tsx`의 문서 일괄등록 미리보기 row에 다음 컬럼을 추가한다.

| 컬럼 | 표시 |
|---|---|
| 선별 | 추천/보류/비추천/검토 필요 pill |
| 점수 | `82점` 형태 |
| 추천 사유 | `fit_reasons` 1~2개 요약 |
| 리스크 | `risk_factors` count 또는 첫 항목 |
| 액션 | 상세 펼침, 등록 선택 체크 |

기본 선택 정책:

- `RECOMMEND` + row `READY`: 기본 선택
- `HOLD`: 기본 미선택 또는 선택 유지 옵션 제공
- `NOT_RECOMMENDED`: 기본 미선택
- `NEEDS_REVIEW`: 기본 미선택
- 사용자가 직접 선택하지 않고 확정 저장하면 백엔드는 `READY` + `RECOMMEND` row만 저장한다.

### 8.2 Row 상세 펼침

각 row를 펼치면 다음 정보를 보여준다.

- 추론된 지원자 기본정보
- 문서별 추출 품질
- 점수 breakdown
- 추천 사유
- 리스크 요인
- 누락 근거
- 면접 확인 포인트
- 공정성 경고

### 8.3 확정 저장 후 상세 화면

`CandidateDetailPage.tsx` / `CandidateDetailModal.tsx` 상세 모드에서는 지원자 기본정보 아래에 `사전 선별 결과` 섹션을 추가한다.

표시 항목:

- 추천 상태
- 점수/신뢰도
- 요약
- 추천 사유
- 리스크
- 면접 검증 포인트
- 결정 상태: 확정/보류/제외

### 8.4 프론트 UI 전략

문서 일괄등록 preview는 “등록 가능 여부”와 “면접 추천 여부”를 분리해서 보여준다.

상단 요약:

- 전체 row 수, 등록 가능 row 수, 추천 row 수, 검토 필요 row 수를 나란히 표시한다.
- 기본 선택 count는 `READY + RECOMMEND` 기준으로 보여준다.
- `HOLD` 이상을 포함해 저장할 수 있는 토글은 2차 옵션으로 둔다. 기본값은 꺼둔다.

테이블:

- 첫 컬럼은 체크박스, 두 번째 컬럼은 지원자명/직무/연락처, 세 번째 컬럼은 선별 pill과 점수로 둔다.
- 선별 pill 색상은 추천=green, 보류=amber, 비추천=gray, 검토 필요=red 계열로 구분한다.
- 점수는 pill 옆에 작게 붙이고, score breakdown은 펼침 영역에서만 보여준다.
- 리스크/누락 근거가 있으면 경고 아이콘과 count를 보여주고 hover 또는 펼침에서 상세를 제공한다.
- `row.status !== READY`는 체크박스를 비활성화하되, 왜 비활성화됐는지 tooltip으로 설명한다.

Row 상세:

- 왼쪽에는 추론된 프로필과 문서 추출 상태를 배치한다.
- 오른쪽에는 추천 사유, 리스크, 누락 근거, 면접 확인 포인트를 탭 또는 접힘 섹션으로 배치한다.
- score breakdown은 운영자가 디버깅할 수 있는 정보라 기본 접힘 상태로 둔다.

확정 저장 CTA:

- 버튼 문구는 `추천 후보 N명 등록`처럼 실제 저장 수를 반영한다.
- 사용자가 `HOLD` 또는 `NOT_RECOMMENDED` row를 선택한 경우 확인 모달에서 “추천 외 후보 포함”을 명시한다.
- 저장 후 toast에는 생성 후보 수, 스킵 수, 오류 수를 표시하고 후보 목록을 갱신한다.

상세 화면:

- 후보자 기본정보 아래에 `사전 선별 결과` 섹션을 추가한다.
- 추천 상태, 점수, 신뢰도, 결정 상태를 한 줄 요약으로 보여주고, 아래에 추천 사유/리스크/면접 확인 포인트를 배치한다.
- `decisionStatus`는 MVP에서는 읽기 전용으로 표시하고, 결정 변경 API가 생기면 액션 버튼으로 확장한다.

## 9. API 변경안

기존 API는 유지하고 response만 확장한다.

| API | 변경 |
|---|---|
| `POST /candidates/document-bulk/preview` | form option `run_screening=true` 추가 가능 |
| `POST /candidates/document-bulk/preview/files` | 동일 |
| `GET /candidates/document-bulk/preview/jobs/{job_id}` | row별 `screening_preview` 포함 |
| `POST /candidates/document-bulk/import` | 저장된 candidate에 screening result 복사 |
| `GET /candidates/{candidate_id}` | 최신 `screening_result` 포함 |

확정 저장 후 별도 조회 API를 둘 수도 있다.

```http
GET /api/v1/candidates/{candidate_id}/screening-results
PATCH /api/v1/candidates/screening-results/{result_id}/decision
```

## 10. 권장 구현 순서

1. `DocumentBulkImportPreviewRow`에 `screening_preview` 필드 추가
2. `_build_preview_row()`에서 휴리스틱 screening score 계산
3. `CandidateDetailModal.tsx` 미리보기 테이블에 추천/점수 표시
4. 추천 row 기본 선택 정책 적용
5. 확정 저장 시 후보자별 `AiJob(job_type=CANDIDATE_RANKING)`에 screening result 복사
6. `CandidateDetailResponse`에 최신 screening result 포함
7. 상세 화면에 사전 선별 결과 섹션 추가
8. 이후 LLM/RAG 기반 보강 노드 추가

## 11. 주의사항

- AI 선별 결과는 자동 불합격이 아니라 HR 검토 보조 정보로 표시한다.
- `birth_date` 등 현재 추출되는 개인정보는 선별 근거로 사용하지 않는다.
- 중복 지원자/추출 실패/필수정보 누락은 “면접 부적합”이 아니라 “등록/검토 리스크”로 분리한다.
- preview 단계에서 LLM 호출이 과도하면 일괄등록 UX가 느려질 수 있으므로, 기본은 휴리스틱, 옵션으로 LLM 보강을 권장한다.
- 확정 저장 전 preview row는 임시 데이터이므로, 나중에 상세 화면에서 보려면 후보자별 `AiJob.result_payload`에 복사해야 한다.
