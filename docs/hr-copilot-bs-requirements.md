HR Copilot BS — 기능 요구 사항 정의서

## 1. FR 코드 구조 및 모듈 정의

| 코드 | 모듈명 | 의미 |
| --- | --- | --- |

| FR-02 | 서비스 사용자 관리 | 일반 사용자 계정 관리 |

## **2. 기능 요구 사항**

### **FR-01 관리자 계정 관리 (`manager`)**

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-01-01 | 관리자 등록 | 기능 | 슈퍼 관리자 계정을 등록한다. 비밀번호는 BCrypt 해시로 저장한다. | 상 |
| FR-01-02 | 관리자 로그인 | 기능 | login_id / password 인증 후 계정 상태를 확인한다. | 상 |
| FR-01-03 | 마지막 로그인 일시 저장 | 기능 | 로그인 성공 시 last_login_at 값을 갱신한다. | 중 |
| FR-01-04 | 사용자 계정 요청 승인 | 기능 | user.request_status가 REQUESTED인 계정을 검토 후 승인/반려 처리한다. | 상 |
| FR-01-05 | 사용자 계정 생성 처리 | 기능 | 승인 시 user.created_by, status, role_type을 설정하여 계정을 활성화한다. | 상 |
| FR-01-06 | 사용자 승인 이력 관리 | 비기능 | 승인 시 approved_by(관리자 ID)를 기록하여 처리 이력을 관리한다. | 중 |

### FR-02 서비스 사용자 관리 (`user`)

> 관리자(manager)에 의해 생성 및 승인되는 서비스 사용자 계정 관리 기능
> 

---

#### 1. 계정 요청 흐름

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-02-01 | 사용자 계정 생성 요청 등록 | 기능 | 요청자 정보(`requested_by_name`, `requested_by_email`)와 요청 사유(`request_note`)를 저장한다. | 상 |
| FR-02-02 | 요청 상태 관리 | 기능 | `request_status`를 `REQUESTED` 상태로 저장한다. | 상 |

---

#### 2. 관리자 승인 흐름

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-02-03 | 사용자 계정 승인 | 기능 | 관리자가 요청된 계정을 승인하고 `request_status`를 `APPROVED`로 변경한다. | 상 |
| FR-02-04 | 사용자 계정 반려 | 기능 | 요청을 반려하고 `request_status`를 `REJECTED`로 변경한다. | 상 |
| FR-02-05 | 승인 이력 저장 | 비기능 | 승인 처리 시 `approved_by`(관리자 ID)를 기록한다. | 상 |

---

#### 3. 계정 생성 및 활성화

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-02-06 | 사용자 계정 활성화 | 기능 | 승인된 계정의 `status`를 `ACTIVE`로 변경한다. | 상 |
| FR-02-07 | 권한 설정 | 기능 | `role_type`을 설정하여 사용자 권한을 정의한다. | 상 |
| FR-02-08 | 생성 관리자 기록 | 비기능 | `created_by`에 계정을 생성한 관리자 ID를 저장한다. | 상 |

---

#### 4. 사용자 관리 기능

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-02-09 | 사용자 목록 조회 | 기능 | 전체 사용자 목록을 조회한다. (`request_status` 포함) | 상 |
| FR-02-10 | 사용자 상세 조회 | 기능 | 특정 사용자 상세 정보를 조회한다. | 상 |
| FR-02-11 | 사용자 정보 수정 | 기능 | 이름, 이메일, 권한(`role_type`)을 수정한다. | 상 |
| FR-02-12 | 사용자 상태 관리 | 기능 | `status`를 `ACTIVE / INACTIVE`로 변경한다. | 상 |
| FR-02-13 | 사용자 삭제 처리 | 비기능 | `deleted_at`, `deleted_by` 기반 논리 삭제를 수행한다. | 상 |
| FR-02-14 | 사용자 접근 제한 | 비기능 | `INACTIVE` 상태 사용자는 로그인 불가 처리한다. | 상 |

| status | `ACTIVE` / `INACTIVE` |

## 3. 테이블 기준 핵심 흐름

| 단계 | 설명 | 관련 테이블 |
| --- | --- | --- |
| 1 | 관리자 또는 사용자 계정 생성 및 로그인 | manager, user |
| 2 | 지원자 등록 및 기본 정보 관리 | candidate |
| 3 | 이력서/포트폴리오 업로드 및 텍스트 추출 | document |
| 4 | 분석 전략 프로파일 선택 | prompt_profile |
| 5 | LLM 호출 및 응답/비용/토큰 로그 기록 | llm_call_log |
| 6 | 생성된 면접 질문 저장 및 관리 | interview_question_item |

## 4. 시스템 핵심 요약

| 구분 | 내용 |
| --- | --- |
| 계정 | 관리자(manager), 사용자(user) 계정을 분리하여 운영 |
| 데이터 중심 | 지원자(candidate)와 문서(document)를 중심으로 분석 수행 |
| AI 전략 | prompt_profile 기반으로 분석 전략을 관리 |
| 결과 | interview_question_item에 질문 결과를 저장 |
| 운영 | llm_call_log로 모델, 응답, 토큰, 비용을 추적 |
