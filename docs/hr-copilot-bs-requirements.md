# HR Copilot BS — 기능 요구사항 정의서

# **전체 테이블 구성**

| 영역 | 테이블 | 설명 |
| --- | --- | --- |
| 관리자 | manager | 관리자 계정 및 인증 |
| 지원자 | candidate | 지원자 정보 관리 |
| 문서 | document | 이력서/포트폴리오 |
| 프롬프트 | prompt_profile | LLM 실행 전략 |
| 실행 | workflow_run | 분석 실행 관리 |
| 결과 | interview_question_item | 면접 질문 |
| 로그 | llm_call_log | LLM 호출 로그 |

# 1. FR 코드 구조 및 모듈 정의

| 코드 | 모듈명 | 의미 |
| --- | --- | --- |
| FR-01 | 관리자 계정 관리 | 슈퍼 관리자 계정 인증 및 상태 관리 |
| FR-02 | 서비스 사용자 관리 | 일반 사용자 계정 관리 |
| FR-03 | 지원자 관리 | 지원자 기본 정보 및 문서 연결 관리 |
| FR-04 | 문서 관리 | 이력서/포트폴리오 업로드 및 텍스트 추출 |
| FR-05 | 프롬프트 관리 | 분석 전략 프로파일 관리 |
| FR-06 | 면접 질문 관리 | 생성 질문 저장 및 수정 |
| FR-07 | LLM 로그 관리 | LLM 호출 이력, 토큰, 비용 추적 |
| FR-08 | 공통 시스템 | Audit, 상태값, 삭제 정책 |

# **2. 기능 요구 사항**

## **FR-01 관리자 계정 관리 (`manager`)**

| 코드 | 기능명 | 구분 | 상세 설명 | 우선순위 |
| --- | --- | --- | --- | --- |
| FR-01-01 | 관리자 등록 | 기능 | 슈퍼 관리자 계정을 등록한다. 비밀번호는 BCrypt 해시로 저장한다. | 상 |
| FR-01-02 | 관리자 로그인 | 기능 | login_id / password 인증 후 계정 상태를 확인한다. | 상 |
| FR-01-03 | 관리자 상태 관리 | 기능 | ACTIVE / INACTIVE / LOCK 상태를 변경할 수 있다. | 중 |
| FR-01-04 | 마지막 로그인 일시 저장 | 기능 | 로그인 성공 시 last_login_at 값을 갱신한다. | 중 |
| FR-01-05 | 관리자 삭제 처리 | 비기능 | 물리 삭제 대신 deleted_at / deleted_by 기반 논리 삭제를 적용한다. | 중 |

| FR-02-02 | 사용자 목록 조회 | 기능 | 관리자가 전체 사용자 목록을 조회한다. | 상 |
| FR-02-03 | 사용자 상세 조회 | 기능 | 특정 사용자의 상세 정보를 조회한다. | 상 |
| FR-02-04 | 사용자 정보 수정 | 기능 | 관리자가 사용자 이름, 이메일, 상태 등을 수정한다. | 상 |

## 3. 테이블 기준 핵심 흐름

| 단계 | 설명 | 관련 테이블 |
| --- | --- | --- |
| 1 | 관리자 또는 사용자 계정 생성 및 로그인 | manager, user |
| 2 | 지원자 등록 및 기본 정보 관리 | candidate |
| 3 | 이력서/포트폴리오 업로드 및 텍스트 추출 | document |
| 4 | 분석 전략 프로파일 선택 | prompt_profile |
| 5 | LLM 호출 및 응답/비용/토큰 로그 기록 | llm_call_log |
| 6 | 생성된 면접 질문 저장 및 관리 | interview_question_item |

## 4. 시스템 핵심 요약

| 구분 | 내용 |
| --- | --- |
| 계정 | 관리자(manager), 사용자(user) 계정을 분리하여 운영 |
| 데이터 중심 | 지원자(candidate)와 문서(document)를 중심으로 분석 수행 |
| AI 전략 | prompt_profile 기반으로 분석 전략을 관리 |
| 결과 | interview_question_item에 질문 결과를 저장 |
| 운영 | llm_call_log로 모델, 응답, 토큰, 비용을 추적 |
