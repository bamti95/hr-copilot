## 6. 식별된 문제점 및 개선 시나리오

### 문제 요약

| #   | 문제                                                       | 영향 노드  | 심각도 |
| --- | ---------------------------------------------------------- | ---------- | ------ |
| P1  | reviewer 레이턴시 과다 (평균 50.8s, P95 71.2s)             | reviewer   | 높음   |
| P2  | questioner 과도한 Input 토큰 (평균 9,672 tokens, I/O=3.45) | questioner | 높음   |
| P3  | 전체 평균 레이턴시 38.6s로 실시간 응답에 부적합            | 전체       | 중간   |
| P4  | 모든 노드에 동일 모델(gpt-5-mini) 적용 — 노드 특성 미반영  | 전체       | 중간   |
| P5  | 노드 간 병렬 실행 미적용 (순차 실행 추정)                  | 전체       | 중간   |

### 개선 시나리오 5종

**시나리오 1: reviewer 출력 제한 및 프롬프트 최적화**

- 현황: 평균 Output 2,999 tokens, 최대 레이턴시 71.2s
- 개선 방향: max_tokens 상한 조정, 구조화된 출력 형식 지정으로 불필요한 장문 생성 억제
- 기대 효과: reviewer 평균 레이턴시 50.8s → 30s 이하 목표, 전체 P95 66.3s → 50s 이하 목표

**시나리오 2: questioner 프롬프트 압축 (Context Trimming)**

- 현황: 평균 Input 9,672 tokens, 전체 Input의 약 39% 차지
- 개선 방향: 대화 히스토리 요약 삽입, 불필요한 시스템 프롬프트 반복 제거, RAG 청크 크기 조정
- 기대 효과: Input 토큰 30% 이상 감소 시 레이턴시 및 비용 동시 개선

**시나리오 3: 노드별 모델 분리 적용 (Model Routing)**

- 현황: 4개 노드 모두 gpt-5-mini 단일 모델 사용
- 개선 방향: driller처럼 단순·반복적 노드는 경량 모델 적용, reviewer처럼 품질이 중요한 노드는 상위 모델 유지 또는 구조화 출력 활용
- 기대 효과: 비용 절감 + 단순 노드 레이턴시 추가 감소

**시나리오 4: 독립 노드 병렬 실행 (Parallelization)**

- 현황: 노드 간 순차 실행으로 추정 (Run당 레이턴시가 단순 합산에 가까움)
- 개선 방향: 의존성 없는 노드(예: driller, predictor)의 병렬 실행 설계
- 기대 효과: Run 단위 end-to-end 소요 시간 단축

**시나리오 5: 레이턴시 이상치 탐지 및 재시도 정책 도입**

- 현황: 60s 초과 호출 8건(10.5%)이 모두 reviewer에서 발생, 현재 재시도 없음
- 개선 방향: 노드별 타임아웃 임계값 설정, 초과 시 경량 폴백 실행 또는 부분 결과 반환 정책 수립
- 기대 효과: 사용자 체감 최대 대기 시간 제어, 서비스 안정성 향상
