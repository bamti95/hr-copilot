# 실험 개요

- **대상:** `jy_questioner` 중심 재시도 경로 및 downstream 노드
- **한 줄 목적:** questioner 출력 압축과 부분 재시도를 통한 전체 토큰 사용량 감소
- **실험 유형**: 비용 / State 구조 개선

---

# 문제 정의 및 관측 데이터

> 이 섹션에서는 "무엇이 문제인지"를 감이 아니라 관측된 현상으로 설명한다.
> 

### 문제 현상

- `questioner` 출력이 길어질수록 이후 노드(`predictor`, `driller`, `reviewer`)의 입력 토큰도 함께 증가했다.
- 재시도 경로에서 낮은 점수 질문이 일부뿐이어도 전체 질문 세트를 다시 처리할 가능성이 있었다.
- `questioner` 산출물의 크기와 재시도 시 State 전달 범위가 downstream 전체에 영향을 주는 구조적 문제가 관측되었다.

# 원인 가설

> 

### 가설 후보

| 번호 | 가설 | 그렇게 생각한 이유 | 확인 방법 |
| --- | --- | --- | --- |
| H1 | `questioner` 출력이 과도하게 길다 | 질문에 포함된 `generation_basis`, `evaluation_guide` 등 필드가 많고 길다 | 질문 수, 문장 수, 필드 길이 제한 후 비교 |
| H2 | 재시도 시 전체 질문을 다시 처리한다 | 낮은 점수 질문 1~2개여도 전체 세트를 재처리 | `target_question_ids` 기반 부분 재시도 적용 |
| H3 | `analyzer` 출력이 너무 길다 | analyzer 결과가 questioner 입력으로 전달되어 토큰 증가 | analyzer 출력 압축 후 비교 |
| H4 | 출력 스키마 자유도가 높다 | 필드별 길이 제한이 없어 모델이 긴 텍스트 생성 | schema에 `max_length` 제약 추가 |

# 실험 설계

> 이 섹션에서는 무엇을 바꾸고, 무엇은 유지했는지 분리해서 기록한다.
> 

### 변경 내용

| 구분 | AS-IS | TO-BE |
| --- | --- | --- |
| 재시도 범위 | 전체 질문을 다시 처리할 가능성 | 낮은 점수 질문 ID만 `target_question_ids`로 유지하고 부분 재처리 |
| State 전달 데이터 | 전체 `answers`, `follow_ups`, `reviews` 재생성 가능 | 기존 결과 유지, 대상 `question_id`만 병합 교체 |
| `questioner` 부분 재생성 | 전체 산출물 초기화 가능 | 전체 산출물 초기화하지 않음 |
| `questioner` 프롬프트 | 후보 수, 문장 수, 필드 길이 제약 약함 | 후보 수, 문장 수, 필드별 길이/개수 상한 명시 |
| QuestionerOutput 스키마 | 길이 제한 부족 | `max_length` 제약 추가 |
| `analyzer` 프롬프트 | 분석 결과 범위가 넓음 | `risks`, `document_evidence` 중심으로 축소 |
| DocumentAnalysisOutput 스키마 | 리스트/문자열 상한 부족 | 리스트/문자열 상한 추가 |
| analyzer 전달 방식 | 원본 결과가 downstream에 전달 | 압축본만 전달 |

### 변인 통제

| 항목 | 유지 / 변경 | 내용 |
| --- | --- | --- |
| 모델 | 🟢 유지 | `gpt-5-mini` 유지 |
| 프롬프트 | 🔴 변경 | `jy_questioner`, `jy_analyzer` 출력 범위와 길이 제한 추가 |
| 테스트 입력 | 🟢 유지 | 동일 계열 HR Copilot 입력 |
| temperature | 🟢 유지 | 기존 설정 유지 |
| max tokens | 🟢 유지 | 명시적 변경 없음 |
| State 구조 | 🔴 변경 | `target_question_ids` 기반 부분 재처리 및 병합 |
| 실행 횟수 | 🟢 유지 | 변경 전후 각 1회 |

### 설계 근거

- `questioner` 출력 토큰을 줄이면 downstream 입력 토큰도 함께 줄어든다.
- `target_question_ids`를 유지하면 재시도 시 필요한 부분만 교체 가능하다.
- analyzer 결과를 압축하면 `questioner`가 처리할 컨텍스트가 줄어든다.

# 회고

### 알게 된 점

- `questioner` 성능은 단일 노드 문제가 아니라 downstream 전체로 누적되는 구조적 문제였다.
- 토큰 감소가 반드시 latency 개선으로 이어지지는 않는다.
- LLM latency는 토큰 수 외에 프롬프트 난도, 출력 형식 제약, 모델 응답 변동성에 영향받는다.
- 재시도 경로에서 부분 병합 방식이 전체 재생성보다 효율적이다.

### 예상과 달랐던 점

- 입력 토큰 28.7% 감소에도 `predictor`, `driller`, `questioner` 실행 시간이 증가했다.
- 모델 내부 reasoning 시간이 토큰 수보다 더 큰 영향을 미칠 수 있다는 점을 확인했다.
- `reviewer` 재시도는 시간이 소폭 감소했으나 전체 워크플로우 개선으로는 이어지지 않았다.

### 추가로 확인해야 할 점

- 동일 입력으로 3~5회 반복 실행해 평균 latency 비교 필요
- `jy_predictor` 노드의 시간 증가 원인 별도 실험 필요
- 비용 로그 집계 방식 점검 ($0.0000 기록 문제)
- 출력 압축이 질문 품질/근거 구체성을 낮추지 않았는지 정성 평가 필요
- 긴 경력기술서 입력에서 질문 개수/필드 길이 제한의 품질 영향 확인

### 한 줄 정리

> LangGraph 기반 HR Copilot에서 `questioner` 출력과 재시도 State 전달량이 downstream 토큰을 증가시킨다는 가설을 세우고, `target_question_ids` 기반 부분 재시도와 analyzer/questioner 출력 압축을 적용하여 전체 토큰 22.9% 감소를 확인했다. 다만 처리 시간이 37.2% 증가해 latency 개선은 실패했고, 비용 및 State 구조 개선만 부분 반영하기로 결정했다.
