# **JY LangGraph 최종 품질 분석 보고서**

> *작성일: 2026-05-07  
목적: `interview_graph_JY`의 Analyzer 기반 문서 분석, `selector_lite` 선별, Predictor/Driller/Reviewer/Scorer 품질 검증 구조 적용 이후 면접 질문 생성 품질이 초기 JY 베이스라인 대비 얼마나 개선되었는지 정량·정성 기준으로 평가하기 위함  
데이터 범위: `backend/ai/interview_graph_JY` 현재 구현, 초기 품질/성능 보고서, 1차·2차 개선 실험일지 기준  
기준: 2026-05-06 JY LangGraph 초기 품질 분석 보고서 및 초기 성능 보고서  
평가 기준: 질문 품질 5항목 + 평가가이드/검증 구조 품질 5항목, 실행 안정성·토큰 효율·재시도 수렴 구조를 함께 반영  
주의: 본 문서는 생성 원문 50문항 수동 채점이 아니라, 현재 JY 파이프라인 구조와 단일 실행 실험 수치에 기반한 최종 품질 분석이다.*

- 현재 JY 그래프는 초기의 단순 멀티 에이전트 순차 구조에서, **문서 리스크 분석 → 후보 질문 생성 → 경량 선별 → 예상답변 → 꼬리질문 → 리뷰 → 점수화 → 최종 선별** 구조로 정리되었다.
- 질문 품질은 단순 문장 자연스러움보다 **문서 근거성, 리스크 검증력, 꼬리질문 연결성, 평가 가능성**을 중심으로 관리된다.

- 전체 LLM 시간은 초기 대비 아직 낮아지지 않았지만, 1차 개선 후 악화된 latency를 2차 개선에서 **424.25초 → 345.62초**로 되돌린 점은 유의미하다.
- 최종 품질 개선의 핵심은 `selector_lite` 자체가 아니라, **품질 검증 전후의 처리 대상을 줄이고, Reviewer/Scorer 결과를 재시도와 최종 선별에 연결한 구조**다.

## **4. 무엇이 실제로 달라졌는가**

### **4-1. 질문 생성의 기준이 바뀌었다**

초기 질문 생성은 Analyzer 결과를 활용하더라도 downstream 전체가 8개 질문을 모두 처리하는 구조였다.

최종 구조에서는 8개 후보를 만들되, 비싼 LLM downstream 전에 `selector_lite`가 먼저 5개 후보로 줄인다. 이후 Predictor, Driller, Reviewer, Scorer는 실제 최종 후보에 가까운 질문을 중심으로 작동한다.

| **초기 질문 처리 성격** | **최종 질문 처리 성격** |
| --- | --- |
| 8개 후보 전체 downstream 처리 | 5개 중심 downstream 처리 |
| 질문 생성 후 품질 검토 | 질문 선별 후 품질 검토 |
| 재시도 시 전체 재처리 가능 | 낮은 점수 질문 중심 부분 재처리 |
| Reviewer 결과 확인 중심 | Reviewer + Scorer 결과를 라우터와 Selector에 반영 |
| 최종 질문 선별만 존재 | 사전 선별 + 최종 선별 이중 구조 |

즉 질문 생성은 **많이 만들고 나중에 고르는 구조**에서 **후보를 만들고, 조기에 줄이고, 검증 결과로 다시 정렬하는 구조**로 이동했다.

### **4-2. Analyzer의 리스크 구조화 효과**

현재 JY 파이프라인에는 별도 `verification_point_extractor` 노드명은 없지만, Analyzer가 유사한 역할을 수행한다.

Analyzer는 다음 정보를 구조화한다.

- 지원자 문서에서 확인되는 리스크
- 문서 근거와 근거의 해석 사유
- 직무 적합성 요약
- 질문화 가능한 검증 포인트

이 결과는 Questioner, Predictor, Driller에 전달되어 다음 효과를 만든다.

- 질문이 일반 경험 확인보다 문서 근거 기반으로 생성된다.
- 예상 답변이 이상적인 모범답안이 아니라 지원자 문서에 근거한 현실적 답변으로 제한된다.
- 꼬리질문이 예상 답변의 빈틈, 역할 범위, 수치, 의사결정, 협업 리스크를 더 직접적으로 찌른다.
- Reviewer와 Scorer가 근거 부족, 직무 불일치, 꼬리질문 약함, 중복 위험을 별도 플래그로 남긴다.

## **5. 대표 구조 비교: 초기 JY vs 최종 JY**

### **5-1. 실행 흐름 변화**

| **구분** | **초기 구조** | **최종 구조** |
| --- | --- | --- |
| 최초 경로 | analyzer → questioner → predictor → driller → reviewer → scorer | analyzer → questioner → selector_lite → predictor → driller → reviewer → scorer |
| 재시도 범위 | questioner 이후 전체 재처리 가능 | 낮은 점수 질문 ID 중심 부분 재생성 |
| 질문 선별 | 최종 selector 1회 | selector_lite + 최종 selector |
| 꼬리질문 생성 | 질문 세트 단위 처리 | 질문 단위 병렬 처리, 동시성 3 |
| 품질 판단 | reviewer/scorer 후 retry 여부 판단 | 품질 플래그별 questioner/driller 재시도 분리 |
| 최종 응답 | 질문 중심 조립 | 질문 + 근거 + 예상답변 + 꼬리질문 + 리뷰 + 점수 조립 |

### **5-2. Scorer 기준 변화**

현재 Scorer는 다음 기준으로 질문별 0~100점 품질 점수를 계산한다.

| **평가 요소** | **반영 방식** | **품질상 의미** |
| --- | --- | --- |
| Reviewer 승인 여부 | approved 기본 38점 | LLM 품질 검토 결과 반영 |
| 문서 근거 존재 | +20점 | 근거 없는 질문 감점 |
| 평가가이드 존재 | +12점 | 면접관 사용성 반영 |
| 역량 태그 존재 | +10점 | 직무 관련성 반영 |
| 리스크 질문 여부 | +10점 | 검증 가치 높은 질문 우대 |
| 예상 답변 존재 | +5점 | 답변 가능성 시뮬레이션 반영 |
| 꼬리질문 존재 | +5점 | 심층 검증 가능성 반영 |
| 중복 질문 | -15점 | 반복 질문 방지 |
| reject_reason 존재 | -10점 | 반려 사유 반영 |

### **5-3. 대표 사례 해석**

초기 구조에서 좋은 질문은 생성될 수 있었지만, 다음 문제가 남아 있었다.

- 어떤 질문이 낮은 점수인지 추적은 가능해도, 재시도 범위가 넓어질 수 있었다.
- 최종 5개만 사용할 예정인데도 8개 전체를 downstream에서 처리했다.
- 질문/가이드 길이가 길어지면 downstream 입력 토큰과 latency가 함께 증가했다.

최종 구조에서는 이 문제가 다음과 같이 완화되었다.

- `selector_lite`가 먼저 후보를 5개로 줄여 downstream 품질 검증 대상을 압축한다.
- 낮은 점수 질문은 최대 2개 ID 중심으로 재생성한다.
- 꼬리질문 약함은 Questioner가 아니라 Driller 재시도로 분리한다.
- 최종 Selector는 승인 여부, 점수, 리스크 질문 여부, 문서 근거 존재 여부를 함께 고려한다.

즉 최종 JY 그래프는 “질문을 생성하는 파이프라인”에서 **질문을 생성하고, 검증하고, 낮은 품질만 보완해 최종 면접 문항으로 정렬하는 파이프라인**으로 바뀌었다.

## **6. 최종 결정**

- **반영 여부:** 반영
- **결정 이유:**
    - 초기 대비 총 토큰이 **100,166 → 65,630 tokens**로 감소해 비용 및 State 전달량 개선 효과가 확인되었다.
    - 1차 개선 후 증가했던 전체 처리 시간이 2차 개선에서 **424.25초 → 345.62초**로 감소했다.
    - `selector_lite` 추가로 최종 5문항만 사용할 구조와 downstream 처리 범위가 정렬되었다.
    - Analyzer가 리스크, 문서 근거, 질문화 포인트를 구조화하고, Questioner가 이를 질문·근거·평가가이드로 변환한다.
    - Predictor/Driller/Reviewer가 분리되어 예상 답변, 꼬리질문, 품질 검토가 한 노드에 섞이지 않는다.
    - Scorer가 Reviewer 결과를 그대로 신뢰하지 않고 문서 근거, 평가가이드, 역량 태그, 꼬리질문, 중복 여부를 규칙 기반으로 다시 점수화한다.
    - Router가 품질 플래그에 따라 Questioner 재시도와 Driller 재시도를 분리해, 재시도 목적이 이전보다 명확해졌다.
    - 최종 응답은 면접관이 바로 사용할 수 있도록 질문, 생성 근거, 문서 근거, 평가가이드, 예상답변, 꼬리질문, 리뷰, 점수를 함께 제공한다.

다만 다음 한계는 남아 있다.

- 생성 원문 50문항 기준 수동 품질 채점은 아직 수행되지 않았다.
- 초기 대비 absolute latency는 아직 낮아지지 않았고, 최종 단일 실행 기준 345초대는 실시간 UX에 부담이 있다.
- 비용 로그가 `$0.0000` 또는 공란으로 기록되는 문제가 있어 실제 비용 분석 신뢰도는 낮다.
- `selector_lite`가 품질 다양성을 낮추지 않는지에 대한 별도 정성 검증이 필요하다.

## **7. 회고**

### **알게 된 점**

- JY 그래프의 품질 병목은 단일 Questioner 프롬프트가 아니라, **질문 후보가 downstream 전체로 어떻게 전달되고 다시 보완되는가**에 있었다.
- 질문 품질은 질문 문장만으로 결정되지 않고, 문서 근거, 예상 답변, 꼬리질문, 리뷰, 점수, 최종 선별이 함께 작동해야 안정된다.
- Reviewer만 두는 것보다 Scorer와 Router를 함께 두는 편이 품질 이슈를 재시도 액션으로 연결하기 쉽다.
- 최종 5개 질문만 쓸 예정이라면 downstream 전체가 8개를 처리하는 구조는 품질보다 비용·시간 낭비를 더 크게 만든다.
- 토큰 감소가 항상 latency 감소로 이어지지는 않지만, 처리 질문 수를 줄이는 구조 변경은 Predictor/Driller 계열 병목 완화에 효과가 있었다.

### **예상과 달랐던 점**

- 1차 개선에서는 토큰이 크게 줄었는데도 latency가 증가했다.
- `jy_predictor`는 입력 길이보다 처리 질문 수와 구조화 출력 개수에 더 민감하게 반응했다.
- `selector_lite`는 품질 노드가 아니라 비LLM 선별 노드지만, downstream 검증 대상을 줄여 전체 품질-성능 균형에 영향을 줬다.
- 재시도는 무조건 많이 할수록 좋은 것이 아니라, 어떤 질문을 고칠지 ID 단위로 제한해야 품질이 흔들리지 않는다.

### **추가로 확인해야 할 점**

- AI_DEV_DATA 10명, 총 50문항 기준으로 질문 원문과 평가가이드를 수동 루브릭 채점해야 한다.
- 최종 `review_summary.avg_score`, `approved_count`, `low_score_count`를 세션별로 수집해 품질 점수 추이를 기록해야 한다.
- SALES, HR, MARKETING 등 다른 직무군에서도 Analyzer 리스크 구조화와 `selector_lite`가 동일하게 효과적인지 검증해야 한다.
- `selector_lite` 선별 전후 질문 다양성, 리스크 질문 비율, 문서 근거 포함률을 비교해야 한다.
- 절대 latency 개선을 위해 Predictor/Driller/Reviewer 병렬화 또는 노드별 모델 라우팅 실험이 필요하다.

## **8. 한 줄 정리**

> *`interview_graph_JY`는 초기의 단순 순차형 멀티 에이전트 질문 생성 구조에서, Analyzer가 문서 리스크와 검증 포인트를 구조화하고, Questioner가 8개 후보를 생성한 뒤 `selector_lite`가 5개로 먼저 줄이며, Predictor·Driller·Reviewer·Scorer가 선별 후보를 검증하는 구조로 개선되었다. 그 결과 초기 대비 총 토큰은 **100,166 → 65,630 tokens**로 **34.5% 감소**했고, 1차 개선 후 악화된 처리 시간도 **424.25초 → 345.62초**로 회복되었다. 최종 구조는 질문 문장 생성보다 **문서 근거, 리스크 검증, 꼬리질문 연결, 품질 점수, 부분 재시도, 최종 선별**을 함께 관리하는 품질 통제형 LangGraph 파이프라인으로 보는 것이 적절하다.*
