# HR Copilot 기본 기능 및 연결성 테스트 체크리스트

구현 상태를 반영한 브라우저 수동 테스트 가이드입니다.  
목적은 "화면에서 보이는 기능이 실제 API/DB와 맞게 연결되어 있는지"와 "지금 바로 고쳐야 할 UX 리스크가 무엇인지"를 빠르게 찾는 것입니다.

## 테스트 범위 메모

- 실제 백엔드 API 연동 확인 가능
  - 인증(Login/Refresh/Logout)
  - 관리자 관리
  - 지원자 관리
  - 지원자 문서 업로드/교체/삭제/다운로드/파싱 결과 조회
  - 프롬프트 프로필 관리
  - 면접 세션 관리
- 현재 프론트 더미 데이터 기반으로 보이는 화면
  - 대시보드
  - 문서 관리
  - 면접 질문 관리
  - 운영 로그

즉, 위 더미 데이터 화면은 "화면/검색/페이징/UI 흐름" 중심으로 테스트하고, "DB 반영/API 응답 규격" 검증 대상에서는 우선순위를 낮게 잡는 것이 맞습니다.

## 테스트 체크리스트

| 구분 | 테스트 시나리오 | 기대 결과 | 확인 방법 | UX/실무 리스크 |
| --- | --- | --- | --- | --- |
| 인증 | 로그인 화면에서 정상 계정으로 로그인 | `/auth/login` 호출 성공, access/refresh token 저장, 관리자 콘솔 첫 화면으로 이동 | 브라우저 Network에서 `POST /api/v1/auth/login` 200 확인, 응답에 `accessToken`, `refreshToken`, `manager` 존재 확인, 새로고침 후 로그인 유지 확인 | 로그인 실패 시 문구가 기술적이면 비전공 HR이 이해하기 어려움. 로그인 버튼 연타 시 중복 요청 체감 여부도 확인 필요 |
| 인증 | 잘못된 ID 또는 비밀번호로 로그인 | 로그인 실패, 로그인 화면 유지, 에러 문구 노출 | 잘못된 계정 입력 후 `POST /api/v1/auth/login` 4xx 확인, 화면 상단/폼 근처 에러 확인 | "무엇이 틀렸는지"는 알려주되 계정 유무를 과하게 노출하면 보안 리스크 |
| 인증 | 로그인 후 새로고침 또는 브라우저 재진입 | 세션 확인 후 자동 로그인 유지 또는 refresh 토큰으로 복구 | 새로고침 시 `checkSession` 동작 확인, access token 제거 후 refresh만 남긴 상태에서 복구되는지 확인 | 세션 확인 중 로딩 화면이 오래 보이면 서비스 장애처럼 느껴질 수 있음 |
| 인증 | 토큰 만료 상태에서 보호 페이지 접근 | 401 발생 시 refresh 시도, 실패하면 로그인 화면으로 이동 | 토큰 조작 후 보호 페이지 진입, `POST /api/v1/auth/refresh` 성공/실패 흐름 확인 | 강제 로그아웃 시 안내 없이 튕기면 사용자 혼란 큼. "세션 만료" 안내 필요 여부 확인 |
| 인증/권한 | `SYSTEM-MANAGER`가 아닌 계정으로 관리자 관리 페이지 진입 후 생성/수정/삭제 시도 | 목록 조회는 가능하되 생성/수정/삭제 UI 또는 동작이 제한되어야 함 | 일반 권한 계정 로그인 후 `/manager/managers` 진입, 등록 버튼/상태 변경/삭제 버튼 동작 확인 | 현재 프론트는 안내문은 있으나 버튼 비활성/숨김이 충분한지 확인 필요 |
| 공통 레이아웃 | 사이드바 메뉴 이동, 모바일 사이드바 열기/닫기, 로그아웃 | 메뉴 이동 정상, 현재 메뉴 active 표시, 모바일에서 오버레이 닫힘 정상 | 데스크톱/좁은 화면에서 직접 클릭, URL과 화면 일치 확인 | 메뉴 상태가 URL과 불일치하면 테스트 중 오판 가능. 모바일에서 닫기 동작 누락 시 사용성 저하 |
| 공통 로딩 | API 호출이 긴 화면에서 전역 로딩 오버레이 표시 | 요청 시작 시 로딩 표시, 완료/실패 시 반드시 해제 | 느린 네트워크로 후보 목록/저장/삭제 수행, 로딩 오버레이 지속 여부 확인 | 로딩이 사라지지 않으면 치명적 blocker. 중복 클릭 방지 여부도 같이 봐야 함 |
| 관리자 관리 | 관리자 목록 조회, 검색, 페이지 이동 | `GET /managers` 결과가 표와 페이징에 반영 | 검색어 입력 후 조회, 페이지 이동 후 결과/전체 건수 확인 | 검색 후 페이지 이동 시 조건 유지되는지 확인 필요 |
| 관리자 관리 | 새 관리자 등록 | `POST /managers` 성공, 목록 재조회 시 신규 계정 노출, DB에 생성됨 | 로그인ID/비밀번호/이름/이메일/권한 입력 후 저장, 응답 201 확인, Swagger 또는 DB 조회로 생성 여부 확인 | 저장 중 버튼 연타 시 중복 생성 가능성 확인 필요 |
| 관리자 관리 | 중복 loginId로 등록 | 409 또는 에러 응답, 화면에 실패 메시지 노출, 중복 생성 안 됨 | 동일 loginId로 재등록 후 Network/DB 확인 | 에러 문구가 서버 원문 그대로면 UX가 거칠 수 있음 |
| 관리자 관리 | 관리자 수정 | `PUT /managers/{id}` 성공, 변경값이 목록/상세 폼에 반영 | 이름/이메일/권한/상태 변경 후 저장, 재조회로 값 확인 | 수정 성공 후 토스트/성공 피드백이 없어 사용자가 저장 여부를 헷갈릴 수 있음 |
| 관리자 관리 | 상태 토글 ACTIVE ↔ INACTIVE | `PATCH /managers/{id}/status` 성공, 목록 badge 즉시 변경 | 상태 변경 후 Network 200, 새로고침 뒤 상태 유지 확인 | LOCKED 상태는 현재 토글 흐름이 단순 ACTIVE/INACTIVE만 고려하는지 확인 필요 |
| 관리자 관리 | 관리자 삭제 | `DELETE /managers/{id}` 후 목록에서 사라지고 DB는 soft delete | 삭제 후 목록 재조회, DB에서 `deleted_at`, `deleted_by` 확인 | 자기 자신 삭제, 마지막 관리자 삭제 같은 운영 리스크 케이스 확인 필요 |
| 관리자 관리 예외 | 필수값 누락, 이메일 형식 오류, 비밀번호 8자 미만, 특수한 loginId 입력 | 프론트 검증 또는 서버 검증으로 저장 차단 | 빈 값/잘못된 형식 입력 후 저장, 에러 위치 확인 | 현재 에러 문구가 필드 근처에 충분히 보이는지 중요 |
| 지원자 관리 | 지원자 목록 조회, 검색, 상태 필터, 페이지 크기 변경 | `GET /candidates` 결과가 표/페이징에 맞게 반영 | 검색어, 상태 필터, 페이지 크기 변경 후 Network params 확인 | 조건 변경 후 현재 페이지가 1로 적절히 초기화되는지 확인 |
| 지원자 관리 | 지원자 신규 등록만 수행 | `POST /candidates` 201 성공, 기본 상태 `APPLIED`, 상세 페이지로 이동 | 필수값 입력 후 저장, 응답 데이터와 화면 상태 확인, DB row 생성 확인 | 저장 직후 상세 이동이 느리면 중복 클릭 유도 가능 |
| 지원자 관리 | 지원자 신규 등록 + 문서 동시 업로드 | 후보 생성 후 문서 업로드가 이어지고, 문서 목록에 `PENDING` 상태로 표시 | 등록 시 파일 1~3개 첨부 후 저장, `POST /candidates` 뒤 `POST /candidates/{id}/documents` 호출 순서 확인 | 생성 성공 후 문서 업로드 실패 시 부분 성공 상태가 생김. 사용자에게 분리 안내 필요 |
| 지원자 관리 | 기존 지원자 기본 정보 수정 | `PUT /candidates/{id}` 성공, 새로고침 후 변경값 유지 | 이름/이메일/전화/생년월일 수정 후 저장, DB 반영 확인 | 변경사항 없을 때도 저장 버튼 누르는 흐름에서 불필요 API 호출 여부 확인 |
| 지원자 관리 | 지원 상태 변경 | `PATCH /candidates/{id}/status` 성공, badge 반영 | APPLIED → SCREENING 등 변경 후 목록/상세 재확인 | 상태 변경 성공 메시지가 없으면 저장 성공 인지가 약함 |
| 지원자 관리 | 지원자 삭제 | `DELETE /candidates/{id}` 후 목록에서 제거, DB는 soft delete | 삭제 후 목록 재조회, DB에서 `deleted_at`, `deleted_by` 확인 | 삭제 확인창만 있고 복구 수단이 없으므로 오삭제 리스크 큼 |
| 지원자 관리 예외 | 중복 이메일 또는 전화번호로 등록/수정 | 저장 실패, 중복 데이터 생성 안 됨 | 같은 이메일/전화번호로 저장 후 400 응답 및 DB 상태 확인 | 서버 메시지가 한글/영문 혼재면 팀 테스트 중 해석 혼선 가능 |
| 지원자 관리 예외 | 이름/이메일/전화번호 비정상 입력 | 프론트 검증 또는 서버 400으로 차단 | 빈 이름, 잘못된 이메일, 자릿수 부족 전화번호 입력 후 저장 | 전화번호 포맷 허용 범위가 넓어도 HR 입장에서는 너무 자유로우면 데이터 품질 저하 |
| 지원자 문서 | 지원자 상세에서 파일 1개 업로드 | `POST /candidates/{id}/documents` 201, 문서 row 생성, 화면 목록 반영 | 이력서 PDF 업로드 후 문서 목록과 DB `document` 테이블 확인 | 업로드 중 진행 상태가 없으면 큰 파일에서 멈춘 것처럼 보일 수 있음 |
| 지원자 문서 | 여러 파일 업로드와 문서 타입 매핑 | 파일 수와 `document_types` 수가 맞게 전달되고 각 문서 타입이 정확히 저장 | 2~3개 파일 선택 후 타입 다르게 지정, Network FormData와 DB row 비교 | 타입 지정 UI가 복잡하면 잘못 매핑된 채 업로드될 수 있음 |
| 지원자 문서 | 문서 업로드 직후 파싱 상태 확인 | 초기 `PENDING`, 이후 background extraction 후 `COMPLETED` 또는 `FAILED`로 바뀜 | 업로드 직후 새로고침 반복, DB `extract_status`와 상세 화면 텍스트 확인 | 자동 새로고침이 없어 사용자가 파싱 완료 시점을 모를 수 있음 |
| 지원자 문서 | 문서 상세 페이지에서 추출 텍스트 확인 | `GET /candidates/{id}/documents/{documentId}` 성공, 텍스트 또는 실패 상태 확인 가능 | 문서 상세 진입 후 본문 확인, DB `extracted_text`와 비교 | 추출 실패 이유가 안 보이면 운영자가 재업로드 판단을 못 할 수 있음 |
| 지원자 문서 | 문서 다운로드 | `GET /download` 응답으로 실제 파일 다운로드 | 다운로드 버튼 클릭 후 파일명/파일 열림 확인 | 다운로드 실패 시 재시도 안내가 없으면 사용자 불만 증가 |
| 지원자 문서 | 기존 문서 교체 | `PUT /candidates/{id}/documents/{documentId}` 성공, 파일 메타데이터 갱신, 추출상태 `PENDING` 초기화 | 새 파일로 교체 후 파일명/크기/상태 변경 확인, 기존 파일 경로 삭제 여부도 확인 | 교체 후 이전 추출 텍스트가 잠깐 남아 보이면 오해 유발 |
| 지원자 문서 | 기존 문서 삭제 | `DELETE /candidates/{id}/documents/{documentId}` 성공, 목록에서 제거, DB soft delete | 삭제 후 목록/DB `deleted_at` 확인 | 삭제 후 로컬 파일까지 제거되므로 오삭제 리스크 큼 |
| 지원자 문서 예외 | 파일 형식 오류, 손상 파일, 0바이트 파일 업로드 | 서버가 실패를 명확히 반환하거나 `FAILED` 처리 | PDF 외 형식, 열리지 않는 파일, 빈 파일로 업로드 후 상태 확인 | 현재 허용 형식/용량 안내가 UI에 부족하면 실사용자가 많이 막힐 수 있음 |
| 지원자 문서 예외 | 3개 초과 파일 추가, 파일 선택 후 타입 미설정, 업로드 중 뒤로가기 | 제한 수량 이상은 차단, 타입 누락 없이 업로드, 중간 이탈 시 상태 일관성 유지 | 파일 4개 선택, 저장 전 뒤로가기, 다시 진입 후 문서 상태 확인 | 현재 프론트는 최대 3개 제한은 있으나 진행 중 취소 UX는 약함 |
| 프롬프트 프로필 | 목록 조회, 검색, 페이지 크기 변경 | `GET /prompt-profiles` 결과가 표/페이징에 반영 | 검색어 입력 후 조회, 페이지 크기 변경 후 Network params 확인 | 검색어 trim 처리로 앞뒤 공백 검색이 무시되는지 확인 |
| 프롬프트 프로필 | 신규 프로필 생성 | `POST /prompt-profiles` 201, profileKey 중복 없이 생성, 목록 반영 | 생성 모달에서 입력 후 저장, 응답/DB 확인 | 생성 모달은 실사용 입력량이 많아 저장 전 미리보기 부족 시 실수 유발 |
| 프롬프트 프로필 | 출력 스키마(JSON) 정상 입력 | 유효 JSON만 저장되고, 재조회 시 동일 문자열 유지 | `{"question":"string"}` 같은 값으로 생성/수정 후 재조회 | JSON 에러를 친절하게 설명하지 않으면 비개발 직군이 막힘 |
| 프롬프트 프로필 | 프로필 수정 | `PUT /prompt-profiles/{id}` 성공, `updated_at` 갱신 | system prompt, output schema 수정 후 저장, 재조회 확인 | 변경 이력/버전 관리가 없어서 운영 중 실수 복구가 어려움 |
| 프롬프트 프로필 | 프로필 삭제 | `DELETE /prompt-profiles/{id}` 성공, 목록에서 제거, DB soft delete | 삭제 후 목록/DB `deleted_at` 확인 | 향후 AI Agent 단계에서 참조 중인 프로필 삭제 시 영향도가 클 수 있음 |
